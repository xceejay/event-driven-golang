One Ride, Six Events

A single ride request triggers a chain of events. Each step is an independent event processed by a different worker. When one step succeeds, it automatically publishes the next. If a step fails, the engine retries it — without affecting other steps. This is event-driven architecture.

How Event Chaining Works

Each step is a separate event. When "Ride Requested" finishes processing, the worker tells the engine "success" and includes a spinoff event — "DRIVER_MATCHED". The engine publishes that as a brand new event, which goes through the same lifecycle: queue, dispatch, process.

If a step fails, only that step retries. If driver matching fails, the engine waits 5 seconds and retries — without affecting any other ride. The retry delay increases with each attempt (5s, then 10s).

Steps are decoupled. The "Trip Started" worker doesn't know about payment. The "Payment" worker doesn't know about drivers. Each is an independent service connected only by events.
